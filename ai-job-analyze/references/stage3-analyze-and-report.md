# 阶段三：分析与报告产出 SOP

## 目标

把 `normalized_jobs.jsonl`、失败项和用户问题转成有判断力、可追溯、设计精致的报告。报告必须优先回答用户输入的分析问题，不做泛泛的岗位市场总结。

默认输出 `report.json` 和 `report.html`。当用户明确要求图片、分享图或配置启用图片导出时，再生成 `report.png`。图片是 HTML/`report.json` 的视觉导出版，不能成为唯一产物。

## 调研依据

- OPM 的 [Job Analysis](https://www.opm.gov/policy-data-oversight/assessment-and-selection/job-analysis/) 观点：岗位分析是招聘、选拔、绩效和职业发展方法的基础。报告要从 JD 中抽取任务、要求、证据和可行动判断，而不是只做词频统计。
- U.S. Department of Labor 的 [O*NET](https://www.dol.gov/agencies/eta/onet?lang=en) 内容模型：岗位可从 knowledge、skills、abilities、work activities、tools/technology 等维度描述。AI 岗位报告应把“职责、能力、技术、业务场景”分开看。
- NACE 的 [career readiness competencies](https://www.naceweb.org/career-readiness/competencies/career-readiness-defined/)：用职业发展、沟通、批判性思维、技术、专业性、协作、领导力等通用能力作为求职侧解释框架，但不要机械套标签。
- USF Career Services 的 [STAR method](https://myusf.usfca.edu/career-services/career-resources/interviews-offers/interviews/star-method)：行动建议要能落到候选人如何组织项目证据，即 situation、task、action、result，而不是只说“提升能力”。
- [Carbon dashboards](https://carbondesignsystem.com/data-visualization/dashboards/)、[USWDS data visualizations](https://designsystem.digital.gov/components/data-visualizations/)、[Atlassian data visualization color](https://atlassian.design/foundations/color-new/data-visualization-color/) 和 [WCAG 2.2](https://www.w3.org/TR/WCAG22/)：报告要有清晰层级、克制配色、常见图表、可访问文本、足够对比度和可读密度。
- OpenAI 的 [GPT Image 2 model page](https://developers.openai.com/api/docs/models/gpt-image-2) 和 [image generation guide](https://platform.openai.com/docs/guides/image-generation)：图片版报告可以通过 Codex 内置 `$imagegen` 或 API/CLI 方式调用 GPT Image 模型生成；当文字和布局必须完全可控时，使用程序化 HTML 渲染截图。

## 分析方法

### 1. JD 信号拆解

对每条归一岗位提取以下信号：

- 角色定位：产品、算法、工程、运营、数据、解决方案、研究等。
- AI 相关度：明确人工智能、人工智能相关、噪声。
- 业务方向：大模型应用、智能体、AIGC、搜索推荐、内容理解、数据策略、平台交付等。
- 能力要求：业务判断、产品规划、技术理解、数据分析、项目推进、跨团队协作、行业理解、表达与文档。
- 技术关键词：模型、RAG、Agent、评测、Prompt、数据平台、推荐、NLP、CV、MLOps 等。
- 资历门槛：校招、初级、中高级、专家、负责人。
- 证据强度：岗位原文是否直接支撑该判断；无法支撑时标为弱证据或未知。

### 2. 求职侧解释框架

把岗位信号翻译成求职者真正关心的问题：

- 机会在哪里：哪些公司、渠道、方向出现了明确需求。
- 门槛是什么：哪些能力是反复出现的硬要求，哪些只是加分项。
- 误区是什么：哪些看起来像 AI 岗，但职责其实不是 AI 核心岗位。
- 如何证明：候选人需要准备什么项目证据、简历表述和面试故事。
- 风险是什么：样本不足、来源失败、城市/渠道偏置、岗位标题噪声。

行动建议必须用 STAR 结构落地：

- Situation：岗位真实业务场景是什么。
- Task：候选人需要证明自己能承担什么任务。
- Action：简历、项目或面试中应该呈现什么动作。
- Result：最好能量化什么结果，不能量化时说明可替代证据。

### 3. 洞察质量标准

每条核心洞察必须满足：

- 有立场：直接说结论，例如“AI 产品岗正在从概念包装转向交付责任”，不要写“市场正在变化”。
- 有证据：关联岗位样本、公司/渠道差异、职责关键词或失败来源。
- 有边界：说明样本范围、失败来源和不能推断的内容。
- 有行动：告诉用户下一步该看什么、补什么、怎么表达。
- 有取舍：指出机会和代价，不写“全面提升综合能力”这类空话。

禁止输出：

- 没有样本支撑的行业判断。
- 只罗列岗位数量和关键词，不解释含义。
- 把所有岗位都包装成高价值机会。
- “建议持续关注”“提升能力”这类无法执行的建议。

## 报告结构

默认 HTML 报告使用以下结构：

1. 标题区：报告主题、公司/渠道/关键词、样本量、生成时间。
2. 一句话结论：最多 35 个中文字符，必须犀利。
3. 关键发现：3 到 5 条，每条包含判断、证据、影响。
4. 样本范围和限制：公司、渠道、岗位数、失败来源、不可推断项。
5. 公司/渠道差异：对比机会、门槛、AI 真实性和求职风险。
6. 能力地图：把岗位要求聚合为能力簇，并标注硬要求/加分项。
7. 岗位画像：列出典型岗位类型、代表职责、适合候选人。
8. 行动建议：用 STAR 结构给简历、项目、面试准备建议。
9. 附录：来源、失败项、字段口径、样本明细入口。

图片报告只保留：

1. 标题和样本范围。
2. 一句话结论。
3. 3 条关键发现。
4. 1 个核心图或能力地图。
5. 3 条行动建议。
6. 来源和限制脚注。

## 组件边界

- 抓取组件只提供原始岗位和失败来源，不直接给结论。
- 字段归一组件只做结构化、去重、分类和证据摘录，不写报告叙事。
- 洞察组件负责回答用户问题，并为每条判断附上岗位样本或统计证据。
- 报告渲染组件只把结构化洞察转成 HTML 或 PNG，不重新解释数据。

## 产物要求

报告阶段必须遵守 `references/output-contract.md`：

- `report.json` 是报告结构化版本，包含结论、证据、图表数据、建议和限制，便于追溯和重渲染。
- `report.html` 是默认必需产物。
- `report.png` 只有用户要求图片或配置启用图片导出时生成。

## HTML 报告流程

1. 读取 `normalized_jobs.jsonl`、`crawl_plan.json`、`crawl_result.json`、用户问题，以及存在时的 `failures.jsonl`。
2. 先生成 `report.json`，把报告结构、图表数据、证据引用和限制说明固化，不要直接写 HTML。
3. 渲染 `report.html`。HTML 必须可独立打开，CSS 内联或本地引用，不依赖远程资源。
4. 自检信息层级、图表是否可读、移动端是否溢出、结论是否有证据。

### HTML 设计规范

- 风格：精致的编辑型数据分析报告，不做花哨营销页。
- 首屏：标题、样本范围、一句话结论、3 个关键数字必须在首屏可见。
- 布局：桌面使用 12 栅格或等价响应式布局；移动端单列；内容宽度控制在 1080 到 1240px。
- 信息密度：每屏只承载一个主问题；每个图表只表达一个核心判断。
- 视觉层级：最重要结论使用最大字号和最高对比度；次级内容降噪。
- 图表：优先使用条形图、排名、分组条形图、矩阵和小倍图；避免复杂雷达图、3D 图和装饰性图。
- 图表标题：标题必须写出洞察，不要只写“岗位分布”。
- 标注：关键峰值、异常值、样本不足必须直接标注在图附近。
- 配色：使用中性色背景、1 个主强调色、1 个风险色、少量辅助色；同一公司或能力簇颜色必须全报告一致。
- 字体：中文优先系统无衬线；正文 15 到 17px，行高 1.55 到 1.75；不要用负字距。
- 可访问性：正文和背景对比度按 WCAG AA 的 4.5:1 目标设计；图表不能只靠颜色传达信息，必要时加标签、纹理或直接文本。
- 附录：保留来源和失败项，避免报告看起来“过度确定”。

### HTML 生成提示词

```text
你是一个犀利但严谨的岗位市场分析师和高级信息设计师。

输入：
- 用户问题：{analysis_question}
- 采集计划：{crawl_plan_json}
- 归一岗位：{normalized_jobs_jsonl}
- 采集失败项：{failures_jsonl}
- 采集结果：{crawl_result_json}

任务：
1. 先生成 report.json，不要直接输出 HTML。
2. 报告必须优先回答用户问题，给出 3-5 条有立场的关键发现。
3. 每条发现必须包含：判断、证据、影响、求职行动建议。
4. 使用岗位原文证据和统计差异，不要补充外部事实。
5. 对样本不足、失败来源、口径偏差要明确写出。
6. 输出 report.html，风格为编辑型数据分析报告，信息量适中、层级清晰、视觉精美。
7. HTML 必须可独立打开，默认不依赖远程资源。
8. 图表标题必须直接表达洞察，不能只写指标名称。
9. 不要输出空泛建议；行动建议必须能转化为简历、项目或面试准备动作。
```

## 图片报告流程

1. 只有用户要求图片、分享图或配置启用图片导出时生成。
2. 先生成或复用 `report.json` 和 `report.html`。
3. 从报告中抽取最关键的 3 条发现、1 个核心图和 3 条行动建议。
4. 按图片生成优先级选择路径，生成 `report.png`。
5. 图片必须保留来源、样本量、生成时间和限制说明。
6. 图片信息必须少于 HTML，不得把完整报告压缩成密集海报。

### 图片生成优先级

1. `$imagegen`：默认优先。适合生成更有设计感的一页式摘要海报、封面图、视觉化信息图。使用时必须输入 `report.json` 摘要、精确文字、样本数字、画布比例和限制说明。生成后必须检查中文、数字、样本范围和结论是否准确；如果文字不稳定，降级到后续路径。
2. API/CLI 直接调用 `gpt-image-2`：当用户明确要求 API、批量、模型控制、成本路径、日志或可复现参数时使用。此路径需要 `OPENAI_API_KEY`，优先使用已安装 `$imagegen` skill 的 `scripts/image_gen.py` CLI，不要临时编写一次性 SDK runner。调用前确认模型、尺寸、质量、输出路径和网络权限。
3. 程序化 HTML 截图：当前两条 AI 图片路径不可用、中文文字必须完全稳定，或版式/图表需要像素级可控时使用。不要用 Python 直接手绘 PNG；使用 `report.json -> Python 生成固定画布 HTML -> Playwright 渲染截图 -> report.png`。

### 程序化图片流程

使用程序化路径时，按以下链路执行：

1. Python 读取 `reports/{run_id}/report.json`，抽取 `headline`、关键发现、行动建议、能力分布、样本数字和限制说明。
2. Python 生成图片专用 HTML：`reports/{run_id}/report-image.html`。
3. `report-image.html` 使用固定画布，默认 `1440x2200`。CSS 内联，不依赖远程资源。
4. 内容必须压缩为一页式摘要海报：1 个结论、3 个关键信号、1 个能力图、3 条求职建议、样本说明。
5. Python 先做内容约束：中文字符默认不超过 450；如果视觉拥挤，继续压缩到 300 字以内。
6. 布局优先保证不裁切：顶部放标题和结论，中段放关键发现和能力图，底部放行动建议、样本范围和限制说明。指标卡优先横向排列，避免底部被裁切。
7. 启动本地 HTTP server，用 Playwright 打开 `report-image.html`。
8. 设置 viewport 为 `1440x2200`，使用 `page.screenshot(...)` 输出 `reports/{run_id}/report.png`。
9. 用 `sips` 或等价工具校验图片尺寸；用 `view_image` 预览，检查裁切、文字溢出、图表可读性和视觉完整性。
10. 如果发现裁切、字数过密、底部元素缺失或中文不可读，先改 HTML/CSS 和文案，再重新截图。

核心模式：

```text
report.json -> Python 生成固定画布 HTML -> Playwright 渲染截图 -> report.png
```

这种方式比 Python 直接用 PIL 画图更适合信息图，因为 HTML/CSS 更容易控制中文排版、卡片、图表、间距和视觉层级。

### 图片设计规范

- 默认定位：高质量一页式洞察海报，而不是完整报告截图。
- 默认画布：竖版长图，推荐 1440x2200；社媒封面可用 1080x1440 或 1080x1920；演示场景可用 1600x900。
- 字号：主标题不小于 48px，正文不小于 28px，脚注不小于 20px。
- 文案：短句、强判断、少段落；单张图不超过 450 个中文字符。
- 图形：最多 1 个主图和 1 个辅助图标组；图表要能在手机上读清。
- 背景：使用浅色或深色稳定底，不使用影响阅读的照片、复杂纹理或装饰性渐变。
- 对比：文本和背景必须有明显对比；避免细线、小字、低透明度文字。
- 归属：底部写明样本范围、数据来源和“图片为摘要，完整报告见 HTML”。

### 图片生成提示词

```text
你是一个高级信息图设计师。基于 report.json 生成一张精美、克制、适合手机阅读的一页式摘要海报。

输入：
- report.json：{report_json}
- 输出比例：{image_ratio_or_default}
- 使用场景：{share_or_presentation_or_user_specified}

任务：
1. 只保留最重要的一句话结论、3 条关键发现、1 个核心图、3 条行动建议。
2. 画面必须精美、克制、信息量适中，适合手机阅读。
3. 不要把完整 HTML 报告截图化；图片是摘要海报。
4. 所有数字、结论和建议必须来自 report.json。
5. 保留样本量、公司/渠道、生成时间和限制说明。
6. 使用清晰视觉层级、稳定配色和大字号；避免低对比、小字、装饰性干扰。
7. 中文和数字必须逐字准确；如果无法保证文字准确，应改用程序化 HTML 截图路径。
```

## 自检清单

- 结论是否直接回答用户问题。
- 每条核心洞察是否有岗位样本、统计差异或失败项支撑。
- 是否说明样本范围、失败来源和不可推断项。
- 是否把“AI 岗位真实性”和“求职证据门槛”讲清楚。
- HTML 是否默认生成，图片是否只在用户要求时生成。
- HTML 是否可独立打开，移动端是否不溢出。
- 图片是否是摘要海报，而不是完整报告截图。
- 报告是否避免空话，行动建议是否能落到简历、项目或面试准备。
